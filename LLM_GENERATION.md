# LLM Authorship Disclosure

This document discloses the LLM authorship of `muskitty-html5-parser`,
in accordance with This Week in Rust's submission policy.

## Summary

The core HTML5 parser logic in this crate was generated by an LLM-based
coding agent (GLM-5.2 via TRAE IDE), supervised by a human developer.
The human wrote the project constitution, the test-feedback loop, and
performed code review; the LLM wrote the Rust implementation.

## What the Human Did

1. **Authored the project constitution** — three hard rules that the
   agent must follow at all times:
   - WHATWG spec is ground truth (spec > WPT > Chromium when they conflict)
   - Safe Rust only (zero `unsafe`, zero C/C++ deps)
   - Spec-compliant, not test-compliant (never modify a test to pass it
     unless the spec proves the test is wrong, with chapter citation)

2. **Built the test-feedback loop** — a Python harness that:
   - Runs `cargo test --test html5lib_tree_construction`
   - Captures per-fixture pass/fail counts
   - Feeds failing test cases back to the agent with the expected
     `#document` output and the relevant WHATWG spec section

3. **Reviewed every commit** — the human reads every diff before
   `git commit`. When the agent takes a shortcut that violates the
   spec (e.g. incorrectly removing an active formatting elements
   marker from `<select>`), the human catches it and requires a
   spec-grounded fix.

4. **Maintained the spec citation discipline** — every behavior change
   must cite a WHATWG section number (e.g. §13.2.6.4.7 step 5). The
   human enforces this in code review.

## What the LLM Did

1. **Implemented the tokenizer** (§13.2.5) — all 85 states of the
   HTML5 tokenizer state machine, including:
   - Content model switching (RCDATA, RAWTEXT, ScriptData, PLAINTEXT)
   - Named character reference resolution (2231 entities, binary search)
   - Numeric character references (decimal + hex)
   - Windows-1252 replacement table
   - Processing instruction and CDATA section states
   - Reentrant design (tree construction can pause/resume tokenizer)

2. **Implemented tree construction** (§13.2.6) — all 21 insertion
   modes, including:
   - Adoption agency algorithm (formatting element cloning)
   - Foster parenting (table context insertion)
   - Foreign content handling (SVG/MathML namespace switching)
   - Template insertion mode with content DocumentFragment
   - Scope checks (default, list, button, table)
   - Reset insertion mode algorithm

3. **Diagnosed and fixed its own bugs** — when the test harness
   reports a failure, the agent reads the failing input, the expected
   output, and the relevant spec section, then proposes a fix. Many
   fixes required multi-step reasoning about the adoption agency
   algorithm's interaction with active formatting element markers.

4. **Selected Rust idioms** — the agent chose `Rc<RefCell<Node>>`
   for DOM nodes (interior mutability + shared ownership), `match`
   on token variants for dispatch, and `enum` for insertion modes.
   These choices were not directed by the human.

## Rust-Specific Observations

These observations are included for TWIR's "Observations/Thoughts"
section, per the policy that tooling updates must have a high amount
of Rust-specific info.

### Borrow Checker Errors as Feedback to the Agent

When the LLM produces code that violates the borrow checker, the
compiler error is fed back directly. The agent typically resolves
the issue in one of three ways:

- **Clone the data** — for cheap-to-clone values like `String` or
  small `Vec`s. The agent learned to prefer this over fighting the
  borrow checker for complex lifetimes.
- **Restructure to avoid the borrow** — e.g. extracting a value
  from a `RefCell` borrow into a local before the next mutable
  borrow, rather than holding the borrow across the call.
- **Use `Rc::clone` instead of references** — for DOM nodes that
  need to be shared across tree operations, the agent consistently
  chose `Rc<RefCell<Node>>` over lifetime parameters.

The agent rarely gets stuck on borrow checker errors for more than
one iteration. The error messages are precise enough that the agent
can usually identify the conflicting borrows and restructure.

### `Rc<RefCell<Node>>` Was the Agent's Choice

The human did not specify the DOM representation. When asked to
implement the DOM Standard's tree operations, the agent chose
`Rc<RefCell<Node>>` for the node type. This is the standard Rust
pattern for shared mutable trees without `unsafe`, but it is
notable that the agent selected it unprompted — likely because it
appears frequently in the agent's training data for Rust DOM-like
structures.

The tradeoff (runtime borrow checks vs. compile-time lifetime
constraints) is acceptable for an HTML parser, where the tree
structure is mutated frequently and the borrow pattern is
data-dependent (not statically known).

### Spec-Guided Codegen Works Well for State Machines

The HTML5 tokenizer is a deterministic state machine with 85 states.
The agent implemented it directly from the WHATWG spec's state
transition descriptions. This is a good fit for LLM codegen because:

- Each state is a small, self-contained function
- The spec describes the transitions in prose, which the agent
  translates to `match` arms
- The test suite (html5lib-tests) provides immediate ground truth

Where the agent struggled was **algorithms with non-obvious
invariants** — the adoption agency algorithm (§13.2.6.4.7) in
particular. The agent initially searched for the "furthest block"
in the wrong direction (bottom-up instead of top-down from the
formatting element), causing index-out-of-bounds panics. The fix
required the human to point to the specific spec step
(§13.2.6.4.7 step 5: "furthest block is the first special element
below the formatting element in the stack of open elements").

### The Constitution Discipline

The "spec > WPT > Chromium" rule turned out to be load-bearing.
When the agent's output disagreed with WPT, the instinct was to
tweak the code to match WPT. The constitution forced the agent to
first check whether WPT or the code was wrong per the spec. In
several cases, WPT's expected output reflected an older spec
version, and the correct fix was to follow the current Living
Standard (which sometimes diverges from WPT). In other cases,
the agent's code was wrong and the spec citation made the fix
unambiguous.

## Test Results

- **Tokenizer**: 99.8% (7022/7036) on html5lib tokenizer suite
  - 14 failures are XML processing instructions, which the spec
    defines as parse errors; the tests expect them to be tokenized
    as comments, which is spec-intentional divergence
- **Tree construction**: 100% (1716/1716) on html5lib tree
  construction suite
  - 204 tests skipped (fragment parsing and script-on mode not
    yet implemented)
